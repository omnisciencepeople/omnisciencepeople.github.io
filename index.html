如菜单、按钮、标签、文本框、滚动条等；
分类:java.awt.Component、java.awt.Menucomponent

寸、位置、颜色等，并实现了一个GUI组件所应具备的基本功能;

容器类本身也是一个组件，具有组件的全部性质，但反过来组件不一定为容器。

Java线程概述2

Java线程创建方式：
一
1.创建一个thread继承类A，并重写A从thread中继承的run方法
2.创建一个A类对象aa
3.调用aa从thread类中继承的start方法（该方法会开辟一个新的线程并自动调用run方法）
二
1.创建一个实现Runnable接口的类A，并重写从Runnable接口继承的run方法
2.创建A类对象aa
3.以aa创建一个thread对象 T 如Thread T=new Thread(aa);
4.调用T的start方法开辟一个新的线程，并自动调用aa中的run方法

注意问题：
执行一个线程实际上就是执行run方法中的代码，直接调用run方法是不会开辟出新的线程的；
执行完aa.start（）后，程序不会立刻执行aa对象所对应的线程，它只是表示aa线程有了可以立即被CPU执行的资格（就绪状
态），但由于想抢占CPU执行的线程有很多，CPU不一定会立即执行aa所对应的线程。
一个thread对象有且只能代表一个线程
一个thread对象不能调用两次start方法，否则会抛出java.lang.IllealThreadState Exception异常
一个thread对象只能创建一个线程
先执行主线程再执行子线程

线程的控制

线程优先级：
线程优先级用数字表示，范围1到10
主线程的缺省优先级为5（默认优先级为5），子线程的优先级默认与父线程相同
创建一个线程默认优先级为5
Java提供一个线程调度器用来监视所有进入就绪状态的所有线程，线程调度器会根据线程优先级决定应该调度哪个线程执行
优先级越高越好
通常高优先级线程将先于优先级低的线程执行，但并不总是这样，因此实际开发中并不单纯依赖优先级来决定线程运行次序。
Thread. NORM_PRIORITY=5
Thread. MIN_PRIORITY=1
Thread. MAX_PRIORITY=10

线程的休眠：
线程的休眠是指：暂停执行当前运行中的线程，使之进入阻塞状态，待经过指定的延迟时间后再醒来并转入就绪状态
Thread中提供的方法

sleep 方法会抛出Interrupted Exception 异常，我们必须对其进行捕捉处理

线程的让步
线程让出CPU，并进入就绪状态
public static void yield()//以类名调用

线程休眠和让步区别：
线程休眠后当前线程进入阻塞状态，转而去执行另一个线程，之后随机执行；
线程让步后成当前线程进入就绪状态，转而去执行另外一个线程，之后交替执行；

线程的串行化
在多线程中，如果在一个线程运行的过程中要用到另一个线程的运行结果，则可进行线程的串行化处理
方法：public final void join() throws InterruptedException

线程的挂起和恢复：
线程的挂起：暂停停止当前运行中的线程，使之转入阻塞状态，并不会自动恢复运行；
线程的恢复：使一个已经挂起的线程恢复运行；
public final void suspend()
public final void resume()
suspend方法挂起线程时并不会释放其锁的资源，这可能会影响其他线程的运行，并且容易导致其他线程的死锁，不提倡使用；
死锁：A和B都相互需要对方的资源，但彼此都守着资源，不给对方使用，导致A和B都无法运行；

线程的同步问题：
经典问题：一张票被多个站台卖
一张票被多个站台卖可能会产生一张票被卖多次的问题；
产生原因：
1.很多代码对同一个资源访问，产生顺序的问题，互斥问题。
2.CPU可以在多个线程的任一点之间来回切换，它们不是互斥的。
互斥（同步）：关键步骤只能由一个线程访问，其他线程不可访问；
以买票为例

假设有三个线程A、B、C。我们知道CPU可在A、B、C任意一个点来回切换；
A发现票数大于0，卖票，但票数未减一就切换到B，B发现票数大于0，卖票，但票数未减一就切换到C，C发现票数大于0，卖票，
票数减一，这样一张票就被卖了三次。CPU在3个线程之间的来回切换可发生在0任意一个点，导致卖票发生紊乱，因为代码1到5
行不是一个整体。
我们希望正常情况：A卖这张票，B、C不可卖，且B、C不可卖下一张票，A卖完这一张票后，下一张票共同竞争，让CPU决定谁卖；
我们引入Synchronized关键词来解决该问题：
Synchronized可用来修饰某个方法、某个方法内部的代码块；
修饰一个方法:
Synchronized void f(){}
修饰一个代码块
Synchronized(对象名)
{
同步代码块
}
Synchronized修饰一个代码块时：
Synchronized(对象名aa)功能：判断aa是否被其他线程霸占，如果发现被其他线程霸占，当前线程陷入等待中。如果发现aa
没有被其他线程霸占，当前线程霸占aa，并执行同步代码块。在当前线程执行期间，CPU可能会切换到其他线程，但其他线程无
法执行同步代码块（因为当前线程霸占了aa对象），当前线程执行完同步代码块后会自动释放对aa霸占，此时他和其他线程共同
竞争对aa的霸占，最终CPU会选择其中的某个线程执行。
导致的结果是一个线程正在操作某资源时，将不允许其他线程操作该资源，即一次只允许一个线程操作该资源。
霸占的专业术语叫锁定，霸占住的对象叫监听器。
Synchronized修饰方法时实际上锁定的是该方法this指针所指向的对象即正在调用该方法的对象；

this.notify()详解：
不是叫醒正在执行this.notify()的线程，而是叫醒一个正在wait this对象的其他线程，如果有多个线程正在wait this对
象，通常叫醒最先wait this对象的线程但具体叫醒哪一个是由系统调度器控制，程序员无法控制。
执行this.notify()方法如果一个线程都没唤醒也是允许的；
this.notifyAll()详解：
唤醒所有因为wait方法而陷入阻塞状态的线程，使之进入就绪状态；
wait方法详解：
会使线程进入阻塞状态，同时会使线程释放对this对象的锁定；

Java线程概述

什么叫程序？
程序是一个严格有序的集合，它规定了计算机完成某一任务时所执行的操作以及这些操作的执行顺序。
单道程序设计环境：
单道程序设计环境是指计算机中除了操作系统以外只存在一个用户程序，即用户程序独享整个计算机资源。
单道程序设计环境特点：
资源独占性：任何时候位于内存中的程序可以使用计算机系统的一切资源，不可能有其他程序与之竞争。
执行顺序性：内存中每次只能有一个程序，各个程序按次执行，绝不可能出现在一个程序运行过程中，又夹杂另一个程序运行的
现象存在，一次只能运行一个程序。
结果的再现性：若运行环境和初始条件相同，同一个程序运行的结果总是一样的。
多道程序设计环境：
多道程序设计环境计算机中除了操作系统外还存在很多用户程序，这些程序同时运行。
多道程序设计环境特点：
间断性：由于资源共享和合作，并发程序间相互制约，造成合作执行间断；
失去封闭性：易受到外界环境影响。
不可再现性：重复执行同一程序时，可能得到不同的结果。因为同一个机器上同时运行多个程序，你永远不知道跟你一起抢占资
源的是哪个程序，过一段时间，运行环境可能就变了。
进程的由来:
一方面为了不改变“ ‘程序’ 是一个在时间上严格有序的指令集合，是静态的保存在存储介质上的”这个概念原有的含义，又能够刻画多个程序共同运行时呈现的新特征，在操作系统中，引入了”进程”这一新的概念。
进程是什么？
进程是程序在某一个数据集上的执行。
进程特征：
动态性：进程是程序的一次执行过程，具有生命周期，能够动态产生和消亡。
并发性：任何进程可以与其他进程并发执行；
独立性：进程是系统进行资源分配和调度的最小单位。
结构性：进程由进程控制块、数据集合、程序三部分组成；
什么是线程？
线程是程序的不同执行路径。
为什么出现线程？
线程能直接共享数据和资源，进程不能。
线程对系统资源冲击比较小，启动比较快，退出比较快。
多重进程不可预期且测试困难。
进程和线程的区别：
进程是系统进行进行资源分配和调度的最小单位；线程是程序执行的最小单位。
一个进程由一个一个或多个线程组成，线程是一个进程中代码执行的不同执行路径。
进程之间相互独立，但同一个进程下的各个线程之间共享程序内存空间及一些进程级资源，某进程内的线程在其他进程不可见。
线程上下文切换比进程上下文切换快得多。
任务调度：
大部分操作系统的任务调度采用的是时间片轮转的抢占式调度方式。
每个线程执行一段时间后强制暂停去执行另一个线程，不论线程的优先级多高，每次执行的时间都是一定的。
优先级高的调度的概率会变大。
任务执行的一段时间称为时间片，任务执行的状态叫做运行状态，任务被暂停处于的状态叫做就绪状态，等待它的下一个时间片
到来。这样每个任务都能执行，由于CPU执行的效率非常高，时间片非常短，在各个任务快速的切换，给人的感觉好像多个任务
在同时执行，这就是并发。
并发和并行：
并发：同一时刻只有一条指令执行，但多个指令被快速轮换，使得在宏观上具有多个指令同时执行的效果。
并行：同一时刻有多个指令同时执行，无论从宏观还是微观来看，二者都是同时执行。
单线程：不过程序多复杂，我们都能用一根线表示，它是有顺序的称这为单线程。
多线程：一个程序的流程在其他地方又单独开辟别的线程，不停占用CPU交替执行。

如何解决哈希冲突？
java利用桶来解决哈希冲突;
map中不存对象本身而是存存储地址，对象存入到后面的线性链表（每个结点存入的是键值对）;

hashmap中当key是用户自定义对象时需要重写hashCode和equals方法;
在HashMap中存储了自定义对象后，要进行取值的过程，会进行hashCode和equals方法的比较，假如不重写，那么使用的就是Object中的hashCode和equals中的方法，都是利用地址值来比较，这样不符合我们的期望，我们更期望的是，当来两个不同的对象，但是他们的属性值是完全一样的，我们认为这是同一个东西，才是我们真正期望的。
